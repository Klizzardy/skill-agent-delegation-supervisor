# Agent Delegation Supervisor — 中文参考指南

## 功能定位

本 Skill 供**监督 Agent** 使用。当用户要求将资料核对、代码检索、开发实现、测试排错、文档整理或受限运维检查交给另一个执行 Agent 时，监督 Agent 负责划定边界、作出高风险决策并验收交付物。本 Skill 提供规则和模板，确保委派过程低 token 消耗、少返工、不越界。

核心原则：**主 Agent 管边界和验收，delegate 管实现。委派的主要实现由 delegate 承担；主 Agent 可直接完成小且明确的验收修复。delegate 不替主 Agent 做架构决策。**

## 适用场景

- 中等等级的实现任务（跨 2-5 个文件，但限于一个子系统内）
- 需要检索多个文件才能完成的调研/定位任务
- 需要写或修改测试但主 Agent 已确认接口签名的任务
- 用户明确要求"交给另一个 Agent 执行"

## 不适用场景

- 单文件、个位数行的修改（主 Agent 直接改更快）
- 跨 5 个以上子系统的架构变更（主 Agent 必须自己掌握全局）
- 涉及密钥、认证配置、CI/CD、部署脚本且用户未明确将其纳入范围的任务（用户明确指定时允许委派，但须限定到指定文件/键并保留未授权内容）
- 加载上下文给 delegate 的 token 成本超过直接做的成本

## 为何能降低 Token 和返工

1. **主 Agent 上下文不膨胀**：delegate 读取和搜索大量代码的过程隔离在子会话中，主 Agent 只需要审阅 diff 和验收报告。
2. **任务书即合约**：SCOPE 字段精确到 glob 模式，delegate 没有借口碰不该碰的文件。
3. **先读后改（Read Before Change）风险分级规则**：阻止 delegate 猜测文件名和接口——这是返工的首要来源。常规范围内的任务，delegate 在一次执行中读取上下文、实现并测试，报告所读事实与 diff/测试证据；涉及配置/部署、公共 API/数据库模式、认证/密钥边界、共享文件冲突或范围歧义的高风险任务，delegate 先只读报告并停下，主 Agent 确认后才允许修改。
4. **四次验收关口**：第 3 号关口（Pre-merge）为通用硬停止点；第 2 号关口（Post-read）对高风险任务为硬停止点，对常规任务整合在 delegate 的单次执行中。问题在最早阶段暴露。

## 精简任务书模板

复制以下模板，填写后随委派一起发送：

```
TASK: [一句话，描述要产出什么，而非怎么产出]
SCOPE:
  - 允许路径: [glob 模式，如 src/guestbook/**/*.py]
  - 禁止路径: [如 config/**, deploy/**, **/*.env, **/credentials*]
  - 绝对不要碰: [明确的文件名或子系统]
上下文文件: [2-5 个文件路径，delegate 应先读取]
只读探索根目录: [仅当准确文件尚未知时填写，由主 Agent 限定允许检索的目录]
现有行为: [1-2 句话描述相关功能的当前行为]
验收标准: [具体、可验证的结果，如 "curl /api/guestbook 返回 200 和 JSON 数组"]
约束条件:
  - 不得修改已有公共函数的 API 签名
  - 未经明确批准不得添加新依赖
  - 所有变更必须保持在允许路径内
```

### 填写要点

- **SCOPE** 写得太宽等于没写。`src/**` 不是有效的 scope——说明主 Agent 没想清楚边界。
- **上下文文件** 选 2-5 个最关键的。太多 delegate 会迷失，太少 delegate 会猜。
- **验收标准** 必须是可执行的。"代码看起来正确"不是验收标准。用 curl 命令、测试命令或具体的 UI 行为描述。

## 委派提示词模板

```
你将收到一个编码任务。严格按以下规则执行：

1. 首先，读取"上下文文件"中列出的每一个文件，报告你发现的关键信息（含文件名和行号）。
2. 基于实际读取的内容（而非猜测）制定修改方案。
3. 只在 SCOPE 允许的路径内进行修改。
4. 完成后，对照验收标准自检，并报告结果。
5. 如果你需要修改 SCOPE 之外的文件或添加新依赖，先停下来报告，等待批准。
6. 风险分级：如果任务涉及用户现有配置/部署文件、公共 API 或数据库模式变更、认证/密钥边界、可能与其他并发任务冲突的共享文件，或 SCOPE 存在歧义——你必须在读取后停下，仅报告所读内容与简要计划，等待主 Agent 确认后再实施。对于范围明确且无上述风险的常规任务，可在同一次执行中完成读取、实现和测试。

以下是任务书：
[粘贴任务书]
```

## 验收报告模板

delegate 完成任务后，应产出一份简短验收报告。主 Agent 用此报告执行第 3、4 号关口：

```
### 修改文件清单
- src/api/routes.py:42-68 — 添加 POST /api/guestbook 端点
- src/models/guestbook.py:15-30 — 新增 GuestbookEntry 数据类

### 未修改但相关的文件（已读取确认不需要改）
- src/db/connection.py — 复用现有连接池，无需修改

### 验收自检
- [x] curl -X POST /api/guestbook -d '{"msg":"test"}' 返回 201
- [x] curl /api/guestbook 返回包含上述消息的 JSON 数组
- [x] 已有测试 src/tests/test_routes.py 全部通过

### 未完成的验收项
- 无
```

## 低 Token 交互规则

当用户明确要求直接执行（如"直接做""开始吧""现在就改"），主 Agent 仍应在内部准备精简任务书，但仅向用户发送一条简短更新后立即启动 delegate：

> "正在委派 [任务] 给实现 Agent，完成后回报结果。"

不要在聊天中展开完整任务书、上下文文件列表或范围细节，除非：
- 任务涉及高风险边界，需要用户决策
- 用户明确要求先查看执行计划

此规则保持用户可见的对话精简，同时维持内部执行的严谨性。

## 典型事故与纠正

### 事故 1：猜测文件名和接口

**表现**：delegate 未读取任何代码便声称"留言墙功能在 `php/guestbook.py`，使用 `/submit` 和 `/list` 接口"。

**根因**：任务未做风险分级；delegate 未收到读取后报告的明确指令，基于经验猜测而非代码事实。

**纠正**：在委派前对任务做风险分级。在第 2 号关口（Post-read）拦截——要求 delegate 先读取上下文文件并报告行号，否则拒绝所有后续输出。对高风险任务强制执行门禁停止；对常规任务中仍出现猜测的，收回任务，主 Agent 自己处理。

同样不得凭空声称仓库状态、配置结构或文件存在。若尚不知道准确文件，应由主 Agent 提供限定目录作为只读探索根目录，不能使用无边界的全盘读取范围。

若连允许探索的项目根目录都尚未确定，主 Agent 应先进行受限发现或向用户取得位置，再委派；不能凭空创造修改范围、提交号或“禁止修改”的文件清单。

### 事故 2：修改未授权配置文件

**表现**：delegate 在实现功能时顺手修改了 `package.json`、`tsconfig.json` 或格式化配置文件。

**根因**：SCOPE 的"禁止路径"没有覆盖这些文件；或 delegate 的格式化工具自动触发修改。

**纠正**：在第 3 号关口（Pre-merge）审查 diff 时逐文件核对是否在 SCOPE 内。不在 SCOPE 内的修改一律拒绝。在任务书的"绝对不要碰"字段中显式列出常见越界文件：`package.json`, `.prettierrc`, `.eslintrc.*`, `tsconfig.json`, `.editorconfig`。

### 事故 3：泄露密钥或敏感信息

**表现**：任务书、提示词或 delegate 输出中包含了数据库连接字符串、API 密钥或内部主机名。

**根因**：主 Agent 在第 1 号关口（Pre-delegation）未检查敏感信息；或将真实值写入了任务书。

**纠正**：在第 1 号关口增加敏感信息检查步骤。严禁将真实密钥、密码或连接字符串写入任务书、提示词或文档。delegate 可在本地读取配置文件以核对结构（键名、类型），但不得在输出中复制真实值。绝不要为了脱敏而改动源配置文件——描述配置结构时使用抽象表述即可。

### 事故 4：无依据声称 Token 节省

**表现**：完成委派后声称"节省了 50% 的 token 消耗"但未做任何计量。

**根因**：主观感受替代了客观测量。

**纠正**：遵循 Token Honesty 原则。如果没有测量，只做定性描述："本次委派使主 Agent 避免了加载 3 个大文件到上下文（约 2000 行代码），主 Agent 只处理了约 150 行的 diff 审阅。"

还要区分两种含义：委派可能减少主 Agent 的额度和上下文负担，但 supervisor 与 delegate 合计的总 token 可能反而更高。没有同时计量两边时，不得宣称总体更省。

若执行 Agent 的运行环境支持 JSON 等结构化用量输出，应对正式委派记录每次调用的输入 token、输出 token、费用和验收结果。不要为很小的文案修正或单点补丁新开一次有高固定上下文开销的会话；将同范围工作合并成一次紧凑委派更划算。

## 发布与使用方式

1. 将本目录（`agent-delegation-supervisor/`）放置于所用 Agent 支持的 skills 目录下，例如 `.codex/skills/` 或 `.claude/skills/`。
2. 主 Agent 在收到符合条件的委派需求时，加载本 Skill。
3. 按 SKILL.md 中的流程执行四次验收关口。
4. 当用户语言为中文时，主 Agent 加载本文件（`references/guide.zh-CN.md`）获取模板和事故参考。
5. 本 Skill 不包含任何项目特定数据，可直接公开发布到 GitHub。
